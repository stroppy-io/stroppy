# Журнал изменений

## Ветка `ydb-internal-tests` (отличия от `main`)

TL;DR:
- обновлена версия YDB Go SDK и связанных пакетов;
- условия вида `WHERE x IN (list)` при выполнении над YDB переведены
  на работу со списком значений для устранения лишних перекомпиляций;
- скорректированы настройки партиционирования таблиц и индексов YDB;
- расширен состав ошибок YDB, для которых выполняется автоматический
  повтор транзакций;
- добавлен опциональный параметр `WAREHOUSE_START` для запуска
  нескольких независимых экземпляров приложения над одной БД;
- ускорена заливка тестовых данных за счёт оптимизации кода генерации
  данных на стороне клиента.

Изменения сгруппированы по подсистемам. Описаны только итоговые отличия
от `main`: правки, отменённые более поздними коммитами этой же ветки, в
описании не отражены.

### Адаптер для YDB

- Поднята версия `ydb-go-sdk/v3` с `3.134.1` до `3.137.0`; вместе с ней
  обновлены сопутствующие модули (`google.golang.org/grpc` до `1.81.1`,
  семейство `go.opentelemetry.io/otel/*` до `1.43.0`, ряд библиотек
  `golang.org/x/*`); версия Go в обоих модулях поднята до `1.25.0`.
- При создании YDB-подключения включена опция `WithLazyTx(true)`:
  `BeginTransaction` теперь откладывается до первого запроса в `sql.Tx`,
  что экономит один RPC к серверам YDB на каждую транзакцию TPC-C.
- Базовый размер пакета `BulkUpsert` увеличен с 2 500 до 5 000 строк.
- Переработана горячая часть `pkg/driver/ydb/insert_spec.go`:
  - введён потоковый `bulkUpsertWriter`, который переиспользует буферы
    строк, типов колонок и `StructValueOption` между пакетами;
  - типы колонок кэшируются между пакетами (`mergeColumnTypes`), и
    повторное сканирование строк ради вывода типов больше не
    выполняется.
- В `pkg/driver/sqldriver/run_query.go` добавлен необязательный
  интерфейс `ArgConverter`, через который диалект может приводить
  именованные параметры к собственным типам до передачи в SQL-драйвер
  (требуется для поддержки списковых параметров в YDB, см. далее).
- Диалект YDB (`pkg/driver/ydb/dialect.go`) реализует `ConvertArg`:
  массивы из JS (`[]any`) приводятся к типизированным `[]int64` или
  `[]string`. Благодаря этому `WithAutoDeclare` выводит параметры как
  `List<Int64>`, текст запроса не меняется от вызова к вызову, и
  попадание в серверный план-кэш устраняет ранее наблюдавшуюся
  постоянную перекомпиляцию для TPC-C.
- Для YDB-запросов TPC-C, использующих `IN`-список
  (`get_items_batch`, `get_stocks_batch`, `stock_count_in`), переход
  на связанный параметр `List<Int64>`: в `workloads/tpcc/ydb.sql`
  плейсхолдеры `IN ({...})` заменены на `IN :item_ids` / `IN :ids`,
  а в `workloads/tpcc/tx.ts` для YDB используется ветка с типизированным
  параметром (для других СУБД сохранена подстановка списка целых
  непосредственно в текст запроса).

### Повторы транзакций YDB (TypeScript)

- В `internal/static/helpers.ts` расширен список шаблонов
  сообщений, по которым `isYDBTransientTxErrorMessage` распознаёт
  временные ошибки и допускает повтор транзакции:
  - `operation/UNAVAILABLE`;
  - `code = 400050`;
  - `wrong_shard_state` (вариант записи через нижний регистр / пробелы)
    в дополнение к уже имеющемуся `WRONG_SHARD_STATE`.
- В тот же файл добавлен комментарий о специфике повторов транзакций
  YDB (срабатывание на тексты gRPC-статусов, экспоненциальная задержка
  через `txRetryPolicy()`).

### TPC-C: партиционирование таблиц YDB

- В `workloads/tpcc/ydb.sql` для всех шардированных по `w_id` таблиц
  (`district`, `customer`, `history`, `new_order`, `orders`,
  `order_line`, `stock`) к существующим
  `AUTO_PARTITIONING_MIN_PARTITIONS_COUNT` добавлен парный
  `AUTO_PARTITIONING_MAX_PARTITIONS_COUNT`, что закрепляет число
  партиций для каждой таблицы.
- Глобальная таблица `item` получила собственный блок
  `AUTO_PARTITIONING_*` (фиксированный диапазон 500–550 партиций).
- Для вторичных индексов `idx_customer_name` (по `customer`) и
  `idx_order` (по `orders`) добавлены отдельные SQL-секции
  `idx_customer_name_setup` / `idx_order_setup`, в которых
  включается автопартиционирование по нагрузке и размеру и задаётся
  фиксированное число партиций (1 000) и целевой размер партиции
  500 МБ.

### TPC-C: запуск нескольких экземпляров над одной БД

- Добавлен сценарий распределённой нагрузки: несколько экземпляров
  Stroppy могут одновременно работать с одной БД, каждый — над своим
  непересекающимся диапазоном складов.
- В `workloads/tpcc/tx.ts` и `workloads/tpcc/procs.ts` появились новые
  переменные окружения:
  - `WAREHOUSE_START` — первый `w_id` для текущего экземпляра
    (по умолчанию `1`);
  - `LOAD_ITEMS` — нужно ли загружать глобальную таблицу `item`
    (по умолчанию `true` при `WAREHOUSE_START=1`, иначе `false`).
- Все спецификации для генератора данных (`warehouse`, `district`,
  `customer`, `stock`, `orders`, `order_line`, `new_order`)
  выдают `w_id` в диапазоне `[WAREHOUSE_START, W_ID_MAX]`.
  `HOME_W_ID` и подбор удалённого склада (`pickRemoteWh`)
  также ограничены этим диапазоном.
- Расчёт ключей партиционирования для YDB (`ydbPartitionKeys`) теперь
  опирается на диапазон текущего экземпляра; в README указано, что
  `create_schema` должен запускаться один раз с глобальными
  `WAREHOUSE_START=1` и полным `WAREHOUSES`.
- Этап `validate_population` фильтрует все агрегаты по своему
  диапазону `w_id BETWEEN WAREHOUSE_START AND W_ID_MAX`; проверки
  по глобальной таблице `item` остались без фильтра.
- В хранимых процедурах `NEWORD` для PostgreSQL и MySQL
  (`workloads/tpcc/pg.sql`, `workloads/tpcc/mysql.sql`) добавлен
  параметр `no_min_w_id`; серверный выбор удалённого склада
  (TPC-C §2.4.1.5) теперь ограничен диапазоном `[no_min_w_id,
  no_max_w_id]`, что сохраняет корректность для распределённого
  прогона. Для одиночного экземпляра поведение не меняется.
- В `workloads/tpcc/README.md` добавлен раздел про распределённые
  прогоны с примерами командных строк и пояснениями про загрузку
  таблицы `item`, схемы и работу `validate_population`.

### Генерация данных: оптимизация по производительности

- Введён пакет переиспользуемых `*rand.Rand`:
  - `pkg/datagen/seed/reusable_prng.go` — `ReusablePRNG` с
    ресидингом одного `rand.PCG` без его пересоздания;
  - `pkg/datagen/seed/prng_pool.go` — общий `sync.Pool`-пул
    `PooledPRNG`, ранее жил в `cmd/xk6air/draw_prng_pool.go`;
    `cmd/xk6air/draw_prng_pool.go` теперь является тонкой обёрткой
    над этим пакетом.
- `pkg/datagen/runtime/context.go` и `pkg/datagen/lookup/lookup.go`
  переведены на `ReusablePRNG`: при каждом `Draw(...)` источник
  ресидится на месте, а не аллоцируется заново.
- ASCII-генератор (`pkg/datagen/expr/kernels.go` и
  `pkg/datagen/expr/stream_draw_text.go`):
  - алфавит, собранный из `AsciiRange`, кэшируется в `sync.Map`
    по отпечатку диапазонов;
  - значения раскладываются в плоскую таблицу `[]byte`
    (или `[]rune`, если хотя бы один codepoint вышел за `0xFF`),
    что превращает выбор символа из O(N ranges) в O(1) индекс;
  - спецслучай `minLen == maxLen` обходит вызов RNG для длины.
- В `cmd/xk6air/draw_arms.go` обращения к источнику изменены с
  `p.r` на экспортированное поле `p.R` после выноса пула в пакет
  `seed`.
